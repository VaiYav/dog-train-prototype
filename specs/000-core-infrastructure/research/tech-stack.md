# Tech Stack Research: DogTrain (базовий продукт)

> Згенеровано: 2026-06-09 · Оновлено: 2026-06-09 (v2 — переорієнтовано на **HTML-прототипи, mobile-first**).
> Мета — стек для **клікабельних HTML-прототипів**, адаптованих під mobile-first застосунок. Прототипи валідують UX, потоки й інструментування метрик **до** побудови повноцінного нативного продукту (відповідає духу завдання: «валідувати гіпотезу, не розробляючи повний функціонал»).

## Що ми будуємо в цьому тест-кейсі
**Не** продакшн-застосунок, а **набір HTML-прототипів**, які виглядають і клікаються як мобільний застосунок (phone-frame ~390px, тач-таргети, безпечні зони). Кожен екран фундаменту (онбординг → профіль → план → «Сьогодні» → прогрес → нагадування) — це прототип, який можна відкрити у браузері й пройти потік.

## Sub-problems to Solve
1. Mobile-first верстка, що виглядає як рідний застосунок (phone-frame, safe-area, тач-таргети ≥44px).
2. Навігація між «екранами» як у застосунку (показ/приховування секцій, hash-роутинг).
3. Збереження стану між екранами/перезавантаженнями (профіль, відмітки, прогрес).
4. Імітація продуктової аналітики (демонструвати схему подій без реального SDK).
5. Імітація локальних нагадувань (UI-банер/Web Notification) — без нативу.
6. Single-file, no-build: відкривається подвійним кліком, працює офлайн.

## Option Comparison

### Sub-problem 1–2: верстка + навігація (mobile-first)
| Опція | Нотатки | Вердикт |
|---|---|---|
| Semantic HTML + CSS (mobile-first, CSS variables, flex/grid) у phone-frame | Без залежностей, повний контроль, легко версіонувати | ✅ Рекомендовано |
| CSS-фреймворк (Tailwind/Bootstrap CDN) | Швидко, але зайва вага для прототипу | ⚠️ Опційно |
| Навігація: vanilla JS show/hide секцій + hash-роутинг | Імітує переходи між екранами застосунку | ✅ Рекомендовано |

### Sub-problem 3: стан
| Опція | Нотатки | Вердикт |
|---|---|---|
| `localStorage` (профіль, відмітки, прогрес) | Стан переживає перезавантаження — реалістична демо | ✅ Рекомендовано (для файлу, відкритого у браузері) |
| In-memory (JS-обʼєкт) | Простіше, але стан губиться при reload | ⚠️ Fallback |

### Sub-problem 4: імітація аналітики
| Опція | Нотатки | Вердикт |
|---|---|---|
| `trackEvent(name, props)` → масив `window.__events` + дебаг-панель/console | Демонструє **точну схему подій** із research, без бекенду | ✅ Рекомендовано |
| Реальний PostHog SDK | Зайве для прототипу; потрібен ключ/мережа | ❌ Поза скоупом прототипу |

### Sub-problem 5: імітація нагадувань
| Опція | Нотатки | Вердикт |
|---|---|---|
| UI-банер «нагадування» в прототипі | Показує патерн без нативу | ✅ Рекомендовано |
| Web Notifications API | Можна для демо, але потребує дозволу браузера | ⚠️ Опційно |

### Sub-problem 6: інтерактивність
| Опція | Нотатки | Вердикт |
|---|---|---|
| Vanilla JS (без фреймворку) | Single-file, нуль білду | ✅ Рекомендовано |
| Alpine.js (CDN, ~15KB) | Трохи реактивності, якщо треба | ⚠️ Опційно |

## Final Recommendation Stack (для прототипів)

| Потреба | Вибір |
|---|---|
| Розмітка/стиль | Semantic HTML + CSS, **mobile-first**, CSS variables, phone-frame ~390px |
| Інтерактив | Vanilla JS (single-file, no-build) |
| Навігація | JS show/hide «екранів» + hash-роутинг |
| Стан | `localStorage` (fallback — in-memory) |
| Аналітика | Імітований `trackEvent()` → лог подій (схема з research) |
| Нагадування | UI-банер (опційно Web Notifications) |
| A/B (демо) | Перемикач варіанта в прототипі (імітує feature flag) |

## Mobile-first вимоги до прототипів
- `<meta name="viewport" content="width=device-width, initial-scale=1">`; layout від 360–390px вгору.
- Тач-таргети ≥44×44px; дії — у зоні великого пальця (низ екрана).
- Жодних hover-only взаємодій; системний шрифт; `prefers-reduced-motion`.
- `env(safe-area-inset-*)`; phone-frame для презентації, але контент адаптивний.
- Контраст ≥4.5:1; стани focus/active видимі.

## Integration Notes
- Схема подій `trackEvent()` **дзеркалить** перелік із `codebase-analysis.md` — це й є «інструментування», яке у проді замінить PostHog.
- Перемикач A/B-варіанта в прототипі демонструє, як виглядатиме flag-gated експеримент Фази 3.

> **Продакшн-нотатка:** реальний DogTrain робився б на RN/Flutter + бекенд + PostHog. Для тест-кейсу ми свідомо валідуємо **HTML-прототипами mobile-first** — дешевше, швидше, і саме для перевірки гіпотез/потоків. Перехід прототип → прод змінює лише реалізацію, не продуктову логіку й не схему подій.
