# UX/UI Patterns: DogTrain (базовий продукт)

> Згенеровано: 2026-06-09 · Джерела: Appcues, NN/g-практики онбордингу, бенчмарки активації, патерни habit/care-апів.

## Core User Flows

### Primary (Happy Path)
```
Сплеш → 3–5 екранів онбордингу (value-first, не «карусель фіч»)
  → Профіль собаки: ім'я, порода, вік, вага, рівень активності, фото
  → Коротке опитування (ціль: здоров'я / активність / поведінка)
  → Генерація плану (з шаблонів) → екран «Сьогодні» з 1–3 першими завданнями
  → ПЕРША ПЕРЕМОГА: відмітка одного завдання (наприклад «перша прогулянка») у першу хвилину
  → Запит дозволу на нагадування У КОНТЕКСТІ (після першої перемоги, не на старті)
```
Ментальна модель на кожному кроці: «Розкажи про собаку → отримай конкретний план → зроби одну дію сьогодні → відчуй прогрес».

### Alternative Paths
- Пропуск фото/ваги (необов'язкові поля) — план усе одно генерується.
- Кілька собак — вибір активного профілю (фундамент: одна собака за замовч., мульти — пізніше).
- Повернення наступного дня — одразу екран «Сьогодні», без повторного онбордингу.

## State Inventory

| Стан | Тригер | Рекомендований патерн |
|---|---|---|
| Empty | Немає завдань на сьогодні / план ще генерується | Дружній empty-стан + 1 запропонована мікродія, ніколи не «порожня сторінка» |
| Loading | Генерація плану / завантаження дня | Скелетон карток (не спінер на весь екран) |
| Error | Збій генерації/мережі | Зрозумілий текст + дані працюють офлайн із локального кешу |
| Success | Завдання відмічено | Миттєвий чек-анімейшн + оновлення прогресу/стріку (оптимістичний UI) |
| Partial | Заповнено не всі поля профілю | План генерується на дефолтах + м'який нудж дозаповнити |
| All-done | Усі завдання дня виконані | Мікросвято (конфеті) + чіткий «що далі / завтра» |

## UI Pattern Library

1. **Value-first онбординг (3–5 екранів):** менше читати — більше робити; перша цінність бажано в межах однієї хвилини.
2. **Екран «Сьогодні» як головний:** денний горизонт із 1–3 картками-завданнями, великі чекбокси (≥44px), кільце/бар прогресу зверху.
3. **Контекстний запит дозволу на пуш:** просити дозвіл **після** першої перемоги й з поясненням цінності (а не системний діалог на старті — це вбиває opt-in rate).
4. **Картка завдання:** іконка типу (прогулянка/тренування/догляд), назва, оцінка часу, «чому це важливо» (1 рядок), чекбокс.
5. **Прогрес без покарання:** стрік/кільце, але пропущений день не «обнуляє все» агресивно (freeze-день).

## Micro-interactions & Animations

- Відмітка завдання → галочка + легка вібрація + інкремент прогресу (оптимістично, до відповіді сервера).
- Завершення всіх завдань дня → конфеті + апдейт стріку (момент «перемоги»).
- Перехід онбординг → «Сьогодні» → плавно, з акцентом на першу дію.

## Accessibility Requirements (WCAG 2.1 AA)

- Тач-таргети ≥44×44px (чекбокси, кнопки).
- Контраст тексту ≥4.5:1; не покладатися лише на колір для статусу (галочка + колір + текст).
- Підтримка динамічного розміру шрифту; екран «Сьогодні» не ламається при великому шрифті.
- VoiceOver/TalkBack: картка завдання озвучує назву + стан («виконано/не виконано»).
- `prefers-reduced-motion` — вимикати конфеті/анімації.

## Platform Considerations (mobile-first)
- Дії в зоні великого пальця (низ екрана).
- Офлайн-first для відміток (локальний кеш → синхронізація): прогулянку відмічають надворі без зв'язку.
- Локальні нотифікації (не лише серверні пуші) — працюють без бекенду й офлайн.

## Anti-patterns to Avoid

- ❌ Системний запит пушів на першому екрані (низький opt-in, корелює з відтоком).
- ❌ Paywall одразу після онбордингу до першої перемоги (конкуренти так роблять — ми ні).
- ❌ Показ усього тижневого плану як стіни завдань (перевантаження — ймовірна причина нашого відтоку, гіпотеза H3).
- ❌ Покарання за пропуск (різке обнулення стріку) — демотивує і жене з апки.
- ❌ Довгий онбординг (>5 екранів) — зростає drop-off до першої цінності.

## Recommended Approach

Фундамент будуємо навколо **одного головного екрана «Сьогодні»** з денним горизонтом і миттєвою «першою перемогою» в онбордингу. Дозвіл на нотифікації просимо контекстно після першої перемоги. Усе інструментуємо аналітикою з першого екрана (Фаза 2 спирається на ці івенти). Тижневий план лишається доступним, але **не є** дефолтним екраном — це прямо знижує перевантаження, яке ми підозрюємо в корені відтоку.
