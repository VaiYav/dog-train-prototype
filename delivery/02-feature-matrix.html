Хто і що може робити одночасно, де залежності, а де треба координуватись. Декомпозиція задач — 01-execution-plan.

⛓ = залежність (послідовно) · ⚠ = спільна поверхня (паралельно, але координувати) · ✅ = незалежні (вільно паралельно)

Як читати: два великі «блокери» — F0 (блокує все) і EN (блокує всі AI). Кластер ⚠ навколо екрана «Сьогодні» — DF·TIP·ADP·TUR (ділять одну поверхню). Решта — ✅ (вільно паралельно).

Хвилі паралельності (що будуємо одночасно)

Wave 0 ──► F0  (фундамент: Today-поверхня, події, сутності, getVariant, rules-engine)
              │  блокує все
              ▼
Wave 1 ──► DF ‖ EN ‖ TUR        ← одночасно, різними руками
              │   (DF↔TUR координують елементи «Сьогодні»)
              ▼  EN розблоковує AI
Wave 2 ──► TIP ‖ ADP ‖ MIS ‖ ASK ‖ LOG   ← усі 5 паралельно
              (TIP·ADP координують «Сьогодні»; усі читають EN read-only)
  • Wave 1 = наші 2 тижні: MVP DF (Dev A+B) + у фоні EN (інші руки) + TUR (не-AI, дизайнер+Dev).
  • Wave 2 = AI-фічі стартують після EN, паралельно між собою (тонкі обгортки).

Хто що веде (ownership × фіча)

Фіча Dev A (frontend/loop) Dev B (data/notif/AI) Designer PT Analyst PT
DF (MVP) стрік/UI/стани нагадування/події/A-B макети/копі A/B+дашборд
EN — збирач контексту — схема даних
TIP картка UI prompt/LLM/кеш тон/копі A/B
ADP прев'ю UI LLM-пропозиція + rules-валідатор — A/B+guardrail
MIS картка (Progress) генерація під ctx формат місій A/B
ASK чат-UI prompt+safety-фільтр копі/дисклеймер A/B+guardrail
LOG поле (Care) parse+підтвердження флоу підтвердження точність парсингу
TUR coach-marks UI події/флаг кроки/копі/a11y A/B (activation)

Точки координації (де фічі «перетинаються»)

Спільне Хто торкається Правило, щоб не зіткнутись
Екран «Сьогодні» DF, TIP, ADP, TUR один порядок слотів (стрік → порада → завдання → адаптив-прев'ю); власник layout — Dev A
Подієва схема усі єдина версіонована схема + тег variant; власник — Dev B + Analyst
Feature flags (getVariant) усі A/B один реєстр флагів; один експеримент = один флаг
PersonalizationContext усі AI стабільний контракт EN; AI читають read-only
rules-engine F0, ADP ADP лише пропонує, валідатор F0 — остання інстанція

Bottom line (для 2 розробників)

  • Зараз (2 тижні): Dev A веде петлю/стрік, Dev B — нагадування+аналітику; паралельно (якщо є люди) — EN у фоні. Це і є валідація однієї гіпотези без повного функціоналу.
  • Далі: як тільки EN готовий — 5 AI-фіч розпаралелюються; TUR (не-AI) можна вставити будь-коли. Жодна AI-фіча не починається до EN — інакше дублюємо збір даних і ловимо неузгодженість.
